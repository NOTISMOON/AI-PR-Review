## 幻灯片 2 · 我们要解决的问题

**痛点**

- 人工 Code Review 耗时、易疲劳，质量随评审人状态波动
- 大型 PR 上下文跨文件，评审人难以一眼看全依赖与影响面
- 初级开发缺乏资深经验，安全 / 性能 / 架构问题容易被漏掉
- 通用 AI 工具只看 diff，缺乏工程化的上下文收集与风险分级

**我们的答案**

> 一个能按需选择审查深度、自动收集跨文件上下文、输出结构化风险报告的 AI 审查平台。

---

## 幻灯片 3 · 产品一句话 & 核心流程

**一句话**：粘贴一个 GitHub PR 链接，选择审查深度，AI 返回带风险等级、置信度和修复建议的结构化报告。

**核心流程（6 步）**

1. 输入 PR URL
2. 选择分析深度（快速 / 标准 / 深度）
3. 配置模型（可选，UI 内配置）
4. AI 分析（流式实时反馈）
5. 查看结果（风险项 + 建议 + 评审评论）
6. 二次审查（可选，对已有结果迭代验证）

---

## 幻灯片 4 · 核心功能一：三级分析模式

> 用一套引擎，三种深度，平衡「速度」与「质量」。

| 模式 | 检查范围 | 上下文 | 适用场景 |
|------|----------|--------|----------|
| **快速 Fast** | 语法/类型错误、明显安全漏洞、运行时错误 | 仅 diff | 快速预检、小型 PR |
| **标准 Standard** | + 逻辑自洽、异常处理、函数签名兼容性 | + 周边代码、依赖图、配置文件 | 日常 PR 审查 |
| **深度 Deep** | + 跨模块依赖、性能热点、架构与可维护性 | + RAG 相关文件、PR 讨论、完整文件 | 重要功能、架构变更 |

逐级累加：深度模式包含标准与快速的全部检查项。

---

## 幻灯片 5 · 核心功能二：二次审查（迭代审查）

> 不是重跑一遍，而是基于上次结论做「再审」。

三种审查动作：

- **验证** —— 核查已识别风险的准确性与严重程度
- **补充** —— 从不同角度发现初次遗漏的问题
- **修正** —— 对已有问题给出更深入分析与替代方案

结构化输出会标注每条结论是「验证通过 / 修正 / 新发现」。在历史页点击「重新分析」即可启用。

---

## 幻灯片 6 · 核心功能三：智能上下文收集

> AI 审查的质量上限，取决于喂给它的上下文。

四种上下文源（按需启用）：

- **依赖图分析**：解析 import/export，构建文件间依赖关系
- **周边代码提取**：补全变更函数的完整定义（而非孤立 diff 片段）
- **相关文件检索（RAG）**：基于变更内容检索语义相关代码
- **配置文件识别**：自动纳入 package.json、tsconfig.json 等

**上下文优先级**（Token 不够时优先保留）：
直接变更 → 被调用函数 → 调用方 → 同文件其他函数 → 外部依赖 → 配置与类型定义

## 幻灯片 7 · 核心功能四：多维度风险评估

**四个审查维度**

- 🛡️ 安全：SQL 注入、XSS、CSRF、密钥泄露
- 🔁 逻辑：边界条件、异常处理、竞态条件
- ⚡ 性能：N+1 查询、内存泄漏、冗余计算
- 🧹 质量：命名规范、代码重复、过度耦合

**双轴评级**（每条风险都带）

- 严重程度：Critical / High / Medium / Low
- 置信度：High（代码直接可见）/ Medium（有证据需确认）/ Low（推测性）

## 幻灯片 8 · 技术亮点：提示词工程

> 不是「把 diff 丢给 GPT」，而是一套模块化、可组合的提示词系统。

**提示词组成**
`基础系统提示词 + 模式专用指令 + CoT 指令 + Few-shot 示例 + 语言特定规则`

**三大设计原则**

1. **职责边界清晰**：每个模式只在其上下文范围内分析，防止过度推断
2. **疑罪从无**：上下文不足时降低风险评级；看不到调用方就假设已正确处理
3. **功能变更识别**：区分「代码缺陷」与「产品决策」，功能变更不计入风险分

## 幻灯片 9 · 技术亮点：多模型支持

> 提供商抽象层设计，无供应商锁定。

**统一接口**
```ts
interface ModelProvider {
  analyze(req): Promise<Result>;
  analyzeStream(req): AsyncIterable<Chunk>;
}
```

**已支持的提供商与模型**

- **OpenAI**：GPT-4o、GPT-4o-mini
- **DeepSeek**：DeepSeek Chat (V3)
- **Anthropic**：Claude Opus 4.8、Sonnet 4.6、Haiku 4.5
- **自定义 / 本地**：任意 OpenAI 兼容 API（Ollama、LM Studio）

用户在 UI 中即可切换模型、配置 API Key 与 GitHub Token。

---

## 幻灯片 10 · 技术亮点：流式响应与工程化

- **SSE 流式输出**：分阶段实时反馈 —— fetching（拉取）→ analyzing（分析）→ validating（校验），风险项逐条推送，无需干等
- **Token 预算管理**：实时估算用量，按优先级动态截断，保证不超模型上下文窗口
- **三层数据校验**：JSON Schema 校验 + 质量检查 + 一致性检查，杜绝 AI 输出格式错误
- **智能缓存**：相同 PR + 模式命中缓存，避免重复消耗
- **双重历史存储**：数据库（Prisma + PostgreSQL）+ 浏览器本地存储，免登录即可查看历史

## 幻灯片 11 · 技术架构总览

```
前端 (Next.js App Router + React + TS)
   │  PRAnalyzer / HistoryPage / AnalysisResults
   ▼
API 层 (Next.js API Routes)
   │  /api/analyze  ── 分析主流程（含二次审查、SSE）
   │  /api/analyses/[id] ── 详情查询
   ▼
核心库 (src/lib)
   ├── prompts/   提示词系统（组装器 + 各模式指令 + Few-shot）
   ├── context/   上下文收集（collector / prioritizer / token-counter / sources）
   ├── models/    模型层（registry / router / provider-factory / providers）
   ├── validation/ 校验（schema / quality / consistency）
   └── cache/     内存缓存
   ▼
数据层
   ├── Prisma ORM → PostgreSQL（云端历史）
   ├── 浏览器本地存储（本地历史）
   └── GitHub API（拉取 PR diff、文件、评论）
```

## 附录 · 演讲备注（口播要点）

- **30 秒电梯陈述**：现有 AI 审查工具大多只看 diff，我们做的是工程化的上下文收集 + 分级审查 + 结构化风险输出，让 AI 真正像资深工程师一样评审 PR。
- **最大差异点**：三级深度 + 二次审查 + 多模型自由切换，三者结合在同类工具里少见。
- **现场演示建议**：用同一个 PR 依次跑「快速」与「深度」，对比风险项数量与上下文丰富度，最直观。
- **被问「准不准」时**：强调「疑罪从无」原则 + 置信度标注 + 三层校验，工具不会乱报，宁可降级也不误判。
